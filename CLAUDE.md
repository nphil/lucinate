# Lucinate — working notes for Claude

Lucinate is a **native SwiftUI (iOS 26, Liquid Glass)** OpenWrt router manager.
The app lives at the repo root (`Lucinate/`, `LucinateTests/`, `project.yml`,
`Support/Info.plist`). The old Flutter app was removed; its history remains in
git and in the `ios-v1.1.x` release tags.

## Project / build

- Xcode project is generated by **XcodeGen** (`project.yml`) — never hand-edit
  a `.pbxproj`. Locally: `xcodegen generate && open Lucinate.xcodeproj`.
- No Swift toolchain in this Linux container — the **macOS CI is the compile
  gate**. Push to a `claude/native-swift-rewrite*` branch to run
  `.github/workflows/ios-native-build.yml` (simulator build + unit tests).
- App icon source is `Icon/*.svg`; `python3 Icon/generate.py`
  (`pip install cairosvg`) renders the 1024 light/dark/tinted PNGs into
  `Lucinate/Resources/Assets.xcassets/AppIcon.appiconset/`. The PNGs are
  committed, so a build needs no icon tooling.

## iOS distribution: Feather / AltStore source

- The repo ships an AltStore/Feather source at `apps.json` (repo root).
  Public link: `https://raw.githubusercontent.com/nphil/lucinate/main/apps.json`
- Format: top-level name/subtitle/description/iconURL/website/tintColor +
  `apps[]`, each app with a `versions[]` list (version, buildVersion, date,
  localizedDescription, downloadURL, size).

## CI/CD rules

- **Develop on a `claude/native-swift-rewrite*` branch; release from `main`.**
  Merging app changes to `main` ships a build.
- `.github/workflows/ios-native-release.yml` — on push to `main` touching the
  app: builds an **unsigned device IPA** (`xcodebuild archive`, no signing),
  versions it `2.0.<github.run_number>` (monotonic so Feather detects updates),
  publishes a GitHub release tagged `ios-v<version>` with a `lucinate.ipa`
  asset, then **patches `apps.json` in-job** (prepends the version, commits
  back with `[skip ci]`).
- **Do NOT rely on a `release: published`-triggered workflow to update
  `apps.json`.** Releases created by the built-in `GITHUB_TOKEN` don't fire
  workflow-triggering events, so such a workflow never runs — the release job
  owns the `apps.json` update itself.
- Releases use the **`ios-v*`** tag namespace.
- Same bundle id as before (`app.cogwheel.lucimobile`) so Feather treats new
  builds as in-place updates.

## After pushing to main — ALWAYS verify the build

Every app push to `main` triggers a real build + public release. After pushing,
confirm (a) the `ios-v<version>` release published with a `lucinate.ipa` asset,
and (b) `apps.json` on `main` gained the new version entry.

Token-efficient method (learned the hard way):

- The build takes several minutes. Wait, then check — don't poll tightly.
- **Use SMALL-output calls:** `mcp__github__get_release_by_tag` /
  `mcp__github__get_latest_release`, and `mcp__github__get_file_contents`
  (apps.json). Do NOT call `list_workflow_runs` — it dumps ~100k+ chars and
  overflows context (if you must, it's saved to a file you can grep/parse).
- Prefer a single delayed direct check (schedule a wakeup ~8 min out) over a
  background subagent for the quick green/red verification.
- For a broken build, pull failing job logs with `get_job_logs`
  (`failed_only`, `tail_lines`) so large log output stays out of context.

### Open thread — router control without joining its WiFi

User asked whether Bluetooth could manage the travel router without connecting
to its WiFi. Conclusion: **not practical** — Lucinate speaks LuCI JSON-RPC →
ubus over HTTP(S) (IP transport only); stock OpenWrt exposes no BLE control
surface, and BT-capable routers (some GL.iNet) only do vendor-proprietary,
per-model BLE provisioning. A real BT path needs a custom GATT↔ubus bridge on
the router firmware — out of scope. The in-scope alternative discussed:
**keep management over the router's WiFi while cellular stays active for
internet** (iOS routes LAN-local over WiFi, internet over cellular when the
WiFi has no usable uplink) — a small app-side change vs. Bluetooth. Also viable:
USB-C→Ethernet into the router LAN, or same-LAN reach in repeater/bridge mode.
Not yet started — waiting on the user to pick a direction.
